# CPT304 Coursework 1：正式选型与「专业版」开发之前的前置工作

**依据：** `CPT304_Coursework/CPT304_Coursework_01_Details.md`（对 CPT304_Coursework_01.pdf 的结构化整理）。

## Problem Frame

官方工作流程为六步：选型 → 审计 → 文献 → 重构（含 5 项基准）→ 报告 → 提交（见作业稿「2.3 工作流程」）。小组在**正式锁定某一仓库并投入「专业合规」重构**之前，若缺少统一认知与可行性验证，容易在后续出现：缺陷选成「基准缺失」、文献与修复脱节、基准无法落地（覆盖率、部署、i18n 等）。本需求文档定义**前置工作**：在步骤 1 书面确认前后必须完成的准备活动与决策产物，但不替代步骤 2–6 本身。

## Requirements

**组织与合规**

- R1. 小组须对齐作业硬性约束：4 人小组、`GroupXX.zip` 交付物、截止日期、Turnitin、AI 使用限制、Learning Mall 提交流程（见作业稿 2.1–2.2）。
- R2. 明确组内角色与「可验证贡献」方式（GitHub 提交、PR 链接），与个人贡献表及评分维度一致（见 2.3.6、2.5.3）。

**认知框架**

- R3. 全员理解「代码缺陷」与「缺失的基准要求」之区别：步骤 2 的 4 个缺陷须为源代码中的独立问题，**不得**将步骤 4 才要求的 i18n、Cookie 横幅、隐私政策等当作审计缺陷（见 4.3 辨析 1）。
- R4. 全员理解最终评分三维：研究质量（50）、基准标准（30）、个人贡献（20），以及报告 8 章结构与字数（见 2.4–2.5）。

** portfolio 与可行性**

- R5. 在从**指定 10 个项目**中做出最终决定前，完成对候选项目的**可比信息收集**：技术栈、本地运行成本、是否便于测试与部署、无障碍/性能的初步印象（见 2.6.1、术语与工具列表）。
- R6. 形成**选型决策记录**：至少包含候选集、淘汰理由、最终项目与 fork 策略（满足步骤 1「从列表 fork」）。

**不纳入前置范围（明确边界）**

- 完成 4 缺陷的正式审计报告级描述、4 篇文献定稿、5 项基准的生产证据——属步骤 2–4，**不作为**本前置工作的完成标准。

## Success Criteria

- 小组能书面回答：**为何选该项目**、**主要风险是什么**、**4 个「真缺陷」可能从哪些模块找**、**5 项基准中哪几项对该栈成本最高**。
- 组长或指定成员能说明：**何时 fork**、**主分支策略**、**谁负责部署/测试/无障碍**，且与作业提交物一致。

## Scope Boundaries

- 不规定具体选题（必须由小组决定）。
- 不包含代写代码、代做文献检索结果；前置工作只要求**流程与模板就绪**。

## Key Decisions

- **前置 = 合规对齐 + 概念澄清 + 多项目可行性扫描 + 选型决策 + 仓库协作约定**；正式「专业版」开发从**步骤 2 审计**与**步骤 4 重构**开始，但前置为二者降低返工风险。
- **可选加速器**：若仓库内已有 `docs/reports/lighthouse-comparative-analysis.md` 等本地 Lighthouse 汇总，可作为 portfolio 扫描的输入之一，**不能替代**小组对目标 fork 仓库的亲自验证。

## Dependencies / Assumptions

- 假设小组能访问 GitHub 与后续部署平台（如 Vercel/Render，见作业稿术语）。
- 假设课程提供的 10 个仓库链接与作业稿一致。
