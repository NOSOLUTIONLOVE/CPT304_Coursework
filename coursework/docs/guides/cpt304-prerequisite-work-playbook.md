# CPT304 Coursework 1：正式选型与专业版开发 — 前置工作手册

**依据：** `CPT304_Coursework/CPT304_Coursework_01_Details.md`（课程 PDF 的结构化整理）。若学校发布的 PDF 与本文冲突，**以官方 PDF / Learning Mall 说明为准**。

本文所称「前置工作」指：在小组**正式 fork 指定列表中的仓库并全面投入重构（步骤 4）**之前，建议完成的**合规对齐、概念澄清、十项目可行性扫描与选型决策**。它不替代官方步骤 2–6，但用于降低返工风险。

---

## A. 小组与合规（对应需求 R1、R2）

| 动作 | 目的 |
|------|------|
| 确认截止日期、4 人小组、`GroupXX.zip` 四件套交付物 | 避免漏交、错命名（见解析稿 2.1–2.2） |
| 全员阅读抄袭、Turnitin、迟交政策 | 风险共知 |
| 对齐 AI 使用边界：仅作语言助手，**不用于核心代码与文献研究** | 与 2.1.2 一致 |
| 约定 Learning Mall 提交人与时间缓冲 | 以平台时间戳为准 |
| 约定可验证贡献方式：Git 提交、PR、分工 | 支撑步骤 6 与个人贡献维度（2.3.6、2.5.3） |

**产出：** 组内一页「规则共识」（可记在会议记录里），无需提交给教师，但应有。

---

## B. 必读概念（对应需求 R3、R4）

### B.1 「代码缺陷」≠「缺失的基准要求」

解析稿 **4.3 辨析 1** 是审计成败关键：

- **可作为步骤 2 的 4 个缺陷：** 源代码中已存在的功能性、结构性、安全性、无障碍等**独立问题**（需用 axe / Lighthouse / 手动检查等**证明**）。
- **不应作为步骤 2 的缺陷：** 作业在步骤 4 才要求的 **5 项最低基准**——例如尚未实现双语言、无 Cookie 横幅、无隐私政策页等。这些应在**重构阶段补齐**，并在报告第 6 章用截图证明，而不是当作「第 2 章发现的缺陷」。

### B.2 评分与报告结构（简表）

| 维度 | 分值 | 证据重心 |
|------|------|----------|
| 研究质量 | 50 | 第 2–5 章：检测 + 文献 + 实现逻辑 + 修复前后代码（解析稿 2.4、2.5.1） |
| 基准标准 | 30 | 第 6 章：5 项基准截图，**图号 + 标题**（2.5.2） |
| 个人贡献 | 20 | 贡献表、GitHub、PR 链接（2.5.3） |

报告总字数 **1500 ±10%**，代码不计入（2.4）。

## C. 十项目可行性扫描（对应需求 R5）

官方列表见解析稿 **2.6.1**（10 个 GitHub 仓库，必须从中 fork，不得自选）。

建议在选型会上讨论（无需一次定案）：

1. **技术栈与组内熟悉度**（静态 / Node / React+Vite / Django 等对测试与部署的影响）。
2. **本地可运行性**（能否在截止前稳定复现问题、写测试）。
3. **步骤 4 的 5 项基准**中，哪几项对该栈**成本最高**（例如 SPA 的测试覆盖率、Django 的 i18n 集成等）——用于预判工期，而非在步骤 2 当作缺陷。
4. **可选输入：** 若本仓库已生成 `docs/reports/lighthouse-comparative-analysis.md`，可将其中无障碍/性能对比作为**旁证**，但仍需在**目标 fork 仓库**上亲自跑 axe / Lighthouse。

**产出：** 填写 [cpt304-project-selection-matrix.md](../templates/cpt304-project-selection-matrix.md)（亦见本文末尾「相关文件」）。

---

## D. 选型决策门（对应需求 R6）

在执行官方 **步骤 1：Project Selection** 前，建议满足：

- [ ] 矩阵中至少 **2 个**候选项目完成「本地跑通 + 粗略工具扫一遍」。
- [ ] 小组能口头列举：若选项目 X，**4 个潜在缺陷可能来自哪里**（例如：某页表单、某组件无障碍），且**不是**「缺 i18n」这类基准项。
- [ ] 指定 **fork 后仓库命名、默认分支、谁开 Group 组织**（避免多人 fork 混乱）。

**正式选型后：** 从列表 **fork** 指定仓库（解析稿 2.3.1），进入步骤 2。

---

## E. Fork 之后、审计之前（与步骤 2 的衔接）

前置工作**结束于**步骤 1 完成且仓库就绪；紧接着进入步骤 2：

1. 使用 axe DevTools、Lighthouse、手动检查等，识别 **4 个具体缺陷**（解析稿 2.3.2）。
2. 文档化「工具 + 现象」，为报告第 2–5 章准备素材。
