# 缺陷识别（步骤 2）：四座「真缺陷」

课程要求：**4 个缺陷 = 源代码里已经存在的独立问题**。  
**不是**缺陷：步骤 4 才要求的双语、Cookie 横幅、隐私政策、达标 Lighthouse 等「基准缺失」，除非能论证这是**实现错误**而非未实现特性。

参考前置说明：`../../coursework/docs/brainstorms/2026-04-21-cpt304-prerequisite-work-requirements.md`（R3）。

## 合格缺陷长什么样

每项应能写清：

1. **可观察现象**（用户或维护者视角）。  
2. **稳定复现步骤**（含测试数据：`DB.txt` 或空载）。  
3. **根因落点**（到「模块/函数/行范围」级别，或明确的 HTML/CSS/JS 片段责任）。  
4. **与作业的独立关联**（为何算 bug，而非产品取舍）。

### 示例方向（须在你 fork 上核实，勿照抄）

单页库存应用常见**可审计**类型包括：

- **数据完整性**：未校验数量/价格为负或极大值；导出 JSON 与界面不一致。  
- **交互/逻辑**：删除/编辑未确认导致误操作；`SAVE` 未处理写入失败。  
- **无障碍**：关键控件仅图标无可访问名称；无 `main` landmark 或 `lang`（若可论证为**错误实现**而非后续 i18n 任务）。  
- **性能/安全**：明显不必要的同步阻塞、或 XSS/注入面（若有用户字符串进 DOM，需证据）。

### 不合格示例

- 「没有英文」→ 双语言是步骤 4 基准，不是步骤 2 独立四缺陷之一，除非原文档另有定义。  
- 「没有 Cookie 横幅」→ 同上。  
- 「Lighthouse 分数低」→ 可作为**线索**，但报告里每座缺陷仍须指向**具体源码或行为**，避免空泛。
