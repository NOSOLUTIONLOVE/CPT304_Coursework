# P1-1「删除确认按钮事件重复绑定」：从《Web/JavaScript 内存泄漏综述》可读出的修法

**缺陷**：同一缺陷在步骤 2 中的编号为 **P1-1**（`../defects/step2-top4-defects.md`）。源码根因在 `app/index.html`：每次打开删除模态时再次 `$('#deleteModal').on('click', '#delete_button', ...)`，导致点击处理器累积。

**综述来源**：`./memory-leak-survey.md`（对应 `./search-queries.md` 中 P1-1 列出的六篇：JSWhiz、Madsen et al.、Arteca et al.、BLeak、LeakPair、Shahoor et al.）。

---

## 1. 综述里与这条缺陷最直接的「方法叙事」

| 来源（综述小节） | 对 P1-1 的含义 |
|------------------|----------------|
| **§2.2 常见泄漏类型——未移除的事件监听器** | 监听器会持续持有闭包与引用链；**重复绑定**等价于在未清理的前提下不断挂载新监听，叠加后既有**功能错误**（一次点击多次触发），也有**滞留引用**的增长风险。这为「必须避免在无清理的情况下重复 `.on()`」提供了文献侧的动机（内存与正确性两用）。 |
| **§6 LeakPair / Shahoor——FP2「未移除的事件监听器 → removeEventListener」** | 工程上强调的修复范式是：**在合适的生命周期点对监听器做对偶清理**。映射到本项目：若在模态或页面维度不能「单次绑定」，则应在再次绑定前**解绑**，或在关闭/销毁时用 **`.off(namespace)` / `removeEventListener`**（视 API 而定）保证至多一份 handler。LeakPair/Shahoor 针对的是框架组件生命周期与 DOM 卸载；你们是 jQuery + Bootstrap Modal，可把「modal 隐藏/关闭」或「委托容器」视作生命周期锚点（见下文实现选项）。 |

其余四篇工作在综述中的角色主要是 **定位、分类、度量泄漏**，而非直接给出与本 bug 一模一样的「点在 modal 里别重复 bind」的配方，但仍可在报告里用来做 **Detection / 论证严重性**：

| 论文 | 从综述可推出的、对「修法选择」的支撑方式 |
|------|------------------------------------------|
| **BLeak（§5）** | **动态**：在「重复打开删除对话框 → 关闭」的循环脚本下观测堆或增长路径；用于说明「重复挂载监听」在真实 SPA 中与可观测泄漏/增长相关联，可作为 **Detection-Literature-Implementation** 中 **Detection** 的深化（工具与方法论），不一定要作为唯一修法依据。 |
| **Madsen et al.（§4.2）** | **静态/理论**：事件调用图（监听边、触发边、时序）强调「注册站点」与「触发站点」的一致性；可用于论证：**在「点击删除行」这一路径上每一次都执行注册**，属于事件资源管理上的不安全模式。 |
| **Arteca et al.（§4.3）** | **模式挖掘**：海量项目中「异常」监听注册对与统计判别；可作支撑句：重复/非常规监听注册在实践中常被标记为高风险用法，与本案「循环内绑定」同属 **API 使用层面的反模式**。 |
| **JSWhiz（§4.1）** | 综述侧重于 **Closure 注解 + 全局赋值等类型化泄漏模式**，与「jQuery 模态内重复 `.on()`」**贴合度较弱**，一般不作为本条缺陷的核心文献论据，除非你们要扩展讨论「全局/长寿命对象持有闭包」。 |

## 2. 可直接写进报告的「方法论 → 本项目实现」对照（不写死唯一方案）

综述与 LeakPair 线给出的共识可压缩为三类 **与文献一致的工程原则**：

1. **注册与卸载对偶**：若监听可能多次挂上，则需有对应的 **单次注册** 或 **显式卸载**（对应 FP2 精神）。
2. **避免在热路径上隐式增殖监听**：监听注册应落在稳定的生命周期边界（初始化一次、委托到稳定父节点），而不是每次用户手势都追加。
3. **可验证**：用固定交互脚本（类比 BLeak 的驱动思路）验证「重复开闭模态后内存或 handler 计数」不因该路径单调恶化。

落实到 **UNIQLO `index.html` 的常见修法**（与 `../defects/step2-top4-defects.md` 已列方向一致；此处仅挂上「综述/文献可读出的理由」）：

| 修法思路 | 与综述的对应话术 |
|---------|-------------------|
| **只绑定一次** `#delete_button`，打开模态时只更新「当前待删行」变量（如 `a`） | 等价于「每个监听目标只监听一次」，从根上杜绝增殖，无需依赖卸载 API。 |
| **绑定前 `.off('click.namespace')` 再 `.on(...)`**，或使用 **`.one('click', ...)`** | 直接体现 FP2 「未移除则补移除 / 或对偶生命周期」的工程化；`.one` 等价于单次消费后不再需要显式移除。 |
| **事件委托**：在 `#deleteModal` 或 `document` 上只做一次 `.on`，用事件目标判断 `#delete_button` | 与「稳定父节点单次注册」一致，避免在 modal 分支里重复挂子元素监听（Madsen 式「谁在何时 listen」更清晰）。 |

## 3. 写步骤 3 / 正文时如何避免「文献与缺陷错位」

- **LeakPair FP2**：最适合支撑「**重复的、未配对清理的事件监听会导致问题**」这一句；在你们个案里重点是 **语义上的「重复注册同一逻辑」**，而不仅是组件卸载遗漏。
- **BLeak**：适合支撑「**为何要在报告里认真对待 SPA 中间路径**」与 Detection 方法论，而不要声称「必须用 BLeak 跑完才算修复」。
- **综述 §2.2**：一段话即可闭环：**监听未移除 → 引用链常驻 → GC 难以回收 → 与用户重复打开 modal 的路径叠加**。再接到你们的 **一次绑定 / `.off` / `.one`** 实现。

## 5. 一句话结论

从这六篇工作的综述中可以**直接支撑**的本项目修法思想是：**把「事件监听」当作须管理生命周期的资源**——遵循 **LeakPair FP2「注册须配套移除或禁止重复挂载」** 与综述 **§2.2「未移除监听器」**；**检测与论证**可选用 **BLeak 式动态复现脚本**、**Madsen/Arteca 式事件与模式层面**的补充论述，而 **JSWhiz** 对本条绑定 bug 的支撑较弱。

若课程要求「每缺陷一篇核心文献」，P1-1 最常与 **LeakPair（ASE 2023）** 或 **Shahoor et al.（TSE 2025）** 的 **FP2** 叙事对齐；具体以你们课程引用格式与导师偏好为准。
