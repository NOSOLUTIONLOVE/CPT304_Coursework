## 1 引言

### 1.1 研究背景

JavaScript 已成为全球最流行的编程语言之一，广泛应用于 Web 前端、服务端（Node.js）以及桌面端（Electron）开发。尽管 JavaScript 具有垃圾回收（GC）机制，但内存泄漏问题依然严重。据研究报道，超过 99% 的 Chrome 在低端 Android 手机上的崩溃由内存问题导致，某些单页应用（SPA）每次交互泄漏高达 186MB 内存。

内存泄漏的根本原因在于：开发者未正确清理对象引用，导致垃圾回收器无法释放已不再使用的内存。常见的泄漏来源包括：未移除的事件监听器、未清除的定时器、未取消的订阅、未取消的动画帧请求等。这些泄漏不会产生运行时错误，也不会导致功能异常，因此极难被发现和调试。

### 1.2 研究意义

随着 React、Vue.js、Angular 等框架的普及，现代 Web 应用越来越复杂，内存泄漏问题也越来越突出。自动化的检测和修复工具对于提升 Web 应用质量和用户体验具有重要意义。本综述涵盖的六篇论文代表了该领域从 2013 年到 2025 年的技术演进，从早期的静态分析方法到最近的主动程序修复技术，展示了该领域的完整发展轨迹。

### 1.3 综述范围

本综述围绕以下六篇代表性论文展开，覆盖了从静态分析到动态检测、从事件驱动编程到内存泄漏修复的全谱：

## 2 背景知识

### 2.1 JavaScript 内存模型

JavaScript 采用自动垃圾回收机制管理内存。GC 通过跟踪对象引用链来判断哪些对象仍然被使用。当一个对象不再被任何"活"引用指向时，它就可以被回收。然而，当对象被"意外"保留——即程序不再需要它，但某个引用仍然存在——就会发生内存泄漏。

> **通俗理解：** 想象一个仓库（内存），GC 是清洁工。正常情况下，清洁工会把没人用的东西扔掉。但如果某个人（引用）一直声称"这东西我还要用"，清洁工就不敢扔。即使这个人已经忘了这东西的存在，它也会一直占着仓库空间。

### 2.2 常见的内存泄漏类型

在 Web/JavaScript 应用中，内存泄漏主要有以下几种形式：

- **未移除的事件监听器**：通过 `addEventListener` 注册的监听器会保持其闭包中所有变量的引用，如果未在组件销毁时调用 `removeEventListener`，这些变量将无法被 GC 回收。

- **未清除的定时器**：`setTimeout`/`setInterval` 返回的 ID 如果未被 `clearTimeout`/`clearInterval` 清除，定时器回调中的所有对象都将保持活跃。

- **未取消的订阅**：在 RxJS 等响应式库中，Observable 被订阅后如果未取消订阅，数据会持续堆积。

- **未取消的动画帧请求**：`requestAnimationFrame` 的递归调用在组件销毁后仍继续执行，保留回调中所有对象。

- **闭包引用**：全局变量（如 `window`）持有对已卸载组件的闭包引用，是最常见的泄漏根源。

### 2.3 事件驱动编程的特殊性

JavaScript 的事件驱动模型给内存泄漏检测带来了独特挑战。事件监听器的注册和触发是解耦的——监听器可能在运行时动态添加，事件名是自由格式的字符串，发射器与监听器之间的关联可以在运行时变化。这意味着传统的静态分析方法难以精确跟踪事件流，而仅依赖运行时信息又会面临可扩展性问题。

## 4 静态分析路线

### 4.1 JSWhiz：基于类型注解的泄漏检测 (CGO 2013)

#### 4.1.1 核心思想

JSWhiz 是该领域最早的系统化工作之一。其核心思想是利用 Google Closure Compiler 的类型注解系统，通过静态类型推断来检测特定模式的内存泄漏。它将内存泄漏视为"类型错误"——当一个对象被赋值给一个类型不匹配的全局变量时，就可能发生泄漏。

> **通俗理解：** 就像检查"把书放进书架"这个动作——如果书架标注的是"放文件"，而你放了一本书，系统就会报警。JSWhiz 就是检查"把对象放进全局变量"这个动作是否类型匹配。

#### 4.1.2 技术方法

JSWhiz 基于程序片段分析（procedural slicing）和指针分析，结合 Google Closure 的类型信息来识别潜在泄漏。其分析流程包括：

1. 解析源代码并提取类型注解
2. 构建程序片段
3. 识别全局变量的赋值模式
4. 检查类型匹配性

如果一个 DOM 元素被赋值给一个期望接收普通对象的全局变量，JSWhiz 就会报告潜在的内存泄漏。

#### 4.1.3 优势与局限

- ✅ **优势：** 全自动、无需运行代码、可集成到编译流程中
- ❌ **局限：** 依赖 Google Closure 类型注解，只能检测特定模式的泄漏，无法处理动态类型的 JavaScript 代码，假正率较高

---

### 4.2 Madsen et al.：事件驱动程序的静态分析 (OOPSLA 2015)

#### 4.2.1 核心创新——事件调用图

这篇论文的核心贡献是提出了"事件调用图"（Event-Based Call Graph）的概念，将传统调用图扩展为包含四种边的图结构：

| 边类型 | 含义 | 回答的问题 |
|:---|:---|:---|
| **调用边 (Call Edge)** | 传统的函数调用关系 | "谁调用了谁" |
| **监听边 (Listen Edge)** | 从 listen 表达式到被注册的监听器函数 | "哪个 listen 注册了哪个函数" |
| **触发边 (Emit Edge)** | 从 emit 表达式到可能被调度的监听器 | "哪个 emit 触发了哪个函数" |
| **May-Happen-Before 边** | 监听器注册与事件触发之间的时序关系 | "注册是否发生在触发之前" |

> **关键洞察：** 一个监听器要能被执行，必须**同时**存在三条边：监听边、触发边和 MHB 边。缺少任何一条都意味着该监听器永远不会被执行（死监听器）。

#### 4.2.2 三种分析设计

论文提出了三种精度递增的分析算法：

| 分析 | 上下文策略 | 精度 | 速度 | 实际效果 |
|:---|:---|:---|:---|:---|
| **基线分析** | 单一上下文，折叠事件循环 | 最低 | 最快 | 43个死监听器警告 |
| **事件敏感分析** | 按已触发事件分离数据流 | 更高 | 指数级最坏情况 | 4/6基准超时 |
| **监听器敏感分析** | 按已注册监听器分离数据流 | 最高 | 中等 | **仅8个警告（减少81%）** |

#### 4.2.3 实验结果

- 在 **12 个 StackOverflow 真实 bug** 程序片段上，三种分析都正确检测到了所有 bug
- 在 **6 个较大的 Node.js 程序**（60-390 行）上，监听器敏感分析在合理时间内将误报减少了 **81%**（从 43 个到 8 个）

### 4.3 Arteca et al.：基于数据挖掘的事件 Bug 模式学习 (TSE 2023)

#### 4.3.1 核心思想——"以量补质"

这篇论文提出了一种独特的策略：不追求精确的静态分析，而是用简单但可扩展的分析从超大规模代码库中挖掘模式，然后用统计方法补偿不精确性。其直觉是：**如果代码库足够大，正确的 API 使用模式会频繁出现，而错误的模式会相对罕见。**

> **通俗理解：** 与其花大力气精确分析每一行代码，不如"广撒网"——分析海量项目，用统计方法找出"异常少数"。就像通过分析百万人的行为来找出"不正常"的那几个。

#### 4.3.2 技术方法

方法分为两个阶段：

**阶段一：数据挖掘**
- 使用 CodeQL 对 **127,531 个**开源项目进行上下文不敏感的静态分析
- 提取事件监听器注册对 `<访问路径, 事件名>`
- 共提取了 **532,004 个**监听器注册对，其中 **160,195 个**唯一对

**阶段二：统计分类**
- 使用基于**二项累积分布函数（BCDF）**的统计检验
- 同时考虑事件名的罕见程度和访问路径的罕见程度
- 将监听器注册对分类为"预期"或"异常"

#### 4.3.3 实验结果

| 指标 | 结果 |
|:---|:---|
| 精确率 | **90.9%**（30个真正例，3个假正例） |
| 召回率 | 7.5% |
| 异常模式数 | 75个，分布在64个项目中 |
| 实际确认 | 向25个项目报告30个issue，**7个已被开发者确认为bug** |

## 5 动态检测路线——BLeak

### 5.1 核心洞察

BLeak 的核心洞察是：**在现代 Web 应用中，用户经常反复回到相同的（近似）视觉状态**（如 Gmail 的收件箱视图）。如果每次回到相同视觉状态时内存持续增长，则是内存泄漏的强烈指示。

> **通俗理解：** 想象你在一家餐厅反复点同样的菜——每次吃完后桌子应该恢复原样。如果每次点菜后桌上都多出几副碗筷不收走，那一定是服务员（GC）漏收了。BLeak 就是自动检测这种"碗筷越堆越多"的现象。

这一观察巧妙地克服了传统"陈旧性"方法在 Web 应用中的根本缺陷——BLeak 发现的泄漏中 **77% 不会被陈旧性方法发现**。

### 5.2 系统架构

BLeak 由三个主要组件构成：

```
┌─────────────────────────────────────────────────────────┐
│                     BLeak Driver                         │
│  启动Chrome → 执行循环脚本 → 获取堆快照 → 运行算法       │
└──────────┬──────────────────────────┬───────────────────┘
           │ Chrome DevTools Protocol │
┌──────────▼──────────┐  ┌────────────▼──────────────────┐
│   BLeak Proxy       │  │        BLeak Agent             │
│  mitmproxy拦截流量   │  │  诊断钩子收集堆栈跟踪          │
│  重写JS代码          │  │  暴露浏览器隐藏状态            │
│  暴露闭包变量        │  │  安装Object/Array/DOM钩子     │
└─────────────────────┘  └───────────────────────────────┘
```

- **BLeak Driver**：启动 Chrome 浏览器，通过 DevTools Protocol 远程控制，执行开发者编写的循环脚本，每轮后获取堆快照
- **BLeak Proxy**：使用 mitmproxy 透明拦截 HTTP/HTTPS 流量，重写 JavaScript 代码以暴露闭包变量
- **BLeak Agent**：嵌入在 Web 应用中的脚本，安装诊断钩子收集增长事件的堆栈跟踪

### 5.3 三大核心算法

#### 5.3.1 泄漏检测算法（PropagateGrowth）

对多个堆快照进行 BFS 遍历，追踪**"路径"而非"对象"**的增长。只有每次快照间都持续增长的路径才会被标记为泄漏根。

> **为什么追踪路径而非对象？** 因为变量或对象属性可能被不断更新为新对象（如 `history = history.concat(newItems)`），追踪路径能处理这种情况。

#### 5.3.2 泄漏诊断算法

通过"钩子"机制诊断泄漏根因：

- **增长钩子**：捕获向泄漏根添加新项时的堆栈跟踪
- **收缩钩子**：从泄漏根移除项时清除对应堆栈跟踪
- **赋值钩子**：泄漏根被赋予新值时重置并重新收集堆栈跟踪

运行一轮循环即可收集所有堆栈跟踪，直接指向导致泄漏的代码。

#### 5.3.3 LeakShare 排名算法

提出了一种新的投资回报率排名指标。与传统指标不同：

| 指标 | 问题 |
|:---|:---|
| **Retained Size** | 只考虑唯一拥有的对象，无法处理共享对象 |
| **Transitive Closure Size** | 由于堆高度连通，大多数泄漏报告相似值 |
| **LeakShare** ✅ | 将共享对象的"功劳"平均分配给所有持有它的泄漏根 |

> **直觉：** 如果一个对象被 4 个泄漏根共同持有，每个根只获得该对象 1/4 的"功劳"。要完全释放该对象，必须修复所有 4 个泄漏根。LeakShare 引导开发者优先修复那些**以最少努力释放最多内存**的泄漏。

### 5.4 实验结果

| 应用 | 泄漏根 | 误报 | 精确率 | 每轮节省 | 增长减少 |
|:---|:---:|:---:|:---:|:---:|:---:|
| Airbnb | 32 | 2 | 94% | 1.04 MB | 81.0% |
| Piwik | 17 | 0 | 100% | 8.14 MB | 99.3% |
| Loomio | 10 | 1 | 90% | 2.83 MB | 98.3% |
| Mailpile | 4 | 0 | 100% | 0.80 MB | 91.8% |
| Firefox Debugger | 4 | 0 | 100% | 0.47 MB | 98.2% |
| **平均** | **67** | **3** | **96.8%** | **2.66 MB** | **93.7%** |

*表 3：BLeak 在五个 Web 应用上的评估结果*

**关键数据：**
- 在开源应用上约 **7 分钟**内完成泄漏定位、排名和诊断
- 中位精确率为 **100%**，平均精确率为 **96.8%**
- 修复后平均消除 **93.7%** 的堆增长
- 共发现 **59 个**之前未知的内存泄漏，涉及 Airbnb、AngularJS、Google Maps SDK、jQuery 等流行库

## 6 自动修复路线

### 6.1 LeakPair：主动程序修复的开创性工作 (ASE 2023)

#### 6.1.1 核心创新——"主动修复"范式

LeakPair 是首个提出"**主动程序修复**"（Proactive Program Repair）概念的工作。与传统"检测→定位→修复"流程不同，LeakPair 的核心洞察是：

> **内存泄漏是非功能性 bug，修复它们不会改变应用行为，因此可以安全地"先修复后验证"，无需先检测泄漏。**

```
传统APR流程：  检测bug → 定位bug → 搜索补丁 → 验证补丁
LeakPair流程： 扫描代码 → 匹配模式 → 应用修复 → 验证非侵入性
```

#### 6.1.2 四种修复模式

LeakPair 通过系统化地挖掘 GitHub 和 StackOverflow 上的真实补丁，提取出四种非侵入式修复模式：

| 模式 | 泄漏类型 | 修复策略 | 覆盖率 |
|:---:|:---|:---|:---|
| **FP1** | 未释放的订阅 | 使用 `takeUntil()` 操作符，在组件析构时发出完成信号 | React 1.6% \| Angular 72.3% |
| **FP2** | 未移除的事件监听器 | 在析构函数中调用 `removeEventListener()` | React 29.8% \| Angular 10.8% |
| **FP3** | 未清除的定时器 | 存储 ID，在析构时调用 `clearTimeout`/`clearInterval` | React 36.3% \| Angular 3.1% |
| **FP4** | 未取消的动画帧请求 | 存储 ID，在析构时调用 `cancelAnimationFrame` | React 14.5% \| Angular 1.5% |

*表 4：LeakPair 的四种修复模式及其覆盖率*

> **关键数据：** 四种模式对已知 SPA 内存泄漏具有很高的覆盖率——React 相关 **82.2%**，Angular 相关 **87.6%**。

#### 6.1.3 修复流程

```
输入：项目源代码
  │
  ▼
┌─────────────────────┐
│  Babel 解析 → AST   │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│  组件检测            │  识别 React/Angular 组件
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│  模式匹配            │  遍历AST，匹配4种泄漏模式
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│  补丁生成            │  在AST中生成修复代码
│  · 避免冗余修复      │  检查是否已有清理代码
│  · 非侵入性检查      │  运行回归测试
└─────────┬───────────┘

          ▼

```

### 7.2 技术演进的关键转折点

通过对比六篇论文，可以识别出以下几个关键转折点：

#### 🔹 从"类型驱动"到"行为驱动"

JSWhiz 依赖类型注解检测特定模式，而 BLeak 通过观察运行时行为（内存增长）来检测泄漏，不再受限于特定模式。这一转变使得检测能力从"已知模式"扩展到"未知模式"。

#### 🔹 从"精确但不可扩展"到"可扩展但精度有损"

Madsen 的精确静态分析只能处理数百行的程序，而 Arteca 通过统计方法实现了对 127K+ 项目的分析，但召回率较低（7.5%）。这揭示了该领域的一个根本性权衡。

#### 🔹 从"只检测"到"检测+修复"

LeakPair 实现了从检测到修复的跳跃，直接输出可应用的修复代码。这一转变将研究目标从"帮助开发者发现问题"提升到"直接解决问题"。

#### 🔹 从"反应式"到"主动式"

传统 APR 需要先检测 bug 再修复，LeakPair 利用内存泄漏的非功能性特点，直接对所有潜在泄漏位置应用补丁。这一范式创新可能推广到其他类型的非功能性 bug。

## 8 开放问题与未来方向

### 8.1 当前面临的共同挑战

1. **可扩展性与精度的权衡：** 精确的静态分析难以处理大型项目，而可扩展的方法往往牺牲精度。如何在保持高精度的同时处理真实规模的 Web 应用仍是一个开放问题。

2. **动态特性的处理：** JavaScript 的动态类型、动态事件名、动态代码执行（eval）等特性给所有分析方法都带来了挑战。

3. **跨文件泄漏：** 当内存的分配和释放分布在不同文件或模块中时，现有工具的处理能力有限。

4. **框架快速演进：** React、Vue、Angular 等框架版本更新快速，修复模式可能迅速过时。

### 8.2 未来研究方向

1. **🤖 LLM 辅助的内存泄漏修复：** 将大语言模型与基于模式的方法结合，利用 LLM 的代码理解能力来发现新的修复模式或处理复杂场景。Shahoor et al. 的 TSE 2025 已开始探索这一方向，但结果表明 GPT-4 在此任务上表现不佳，未来可能需要更专业的微调模型。

2. **🔄 实时监控与预警：** 将内存泄漏检测集成到 CI/CD 流水线中，实现开发过程中的实时监控和预警。

3. **🌐 跨平台支持：** 将现有方法扩展到桌面端（Electron）、移动端（React Native）等更多平台。

4. **⚙️ 自动化模式挖掘：** 将 LeakPair 的手动模式挖掘过程自动化，实现修复模式的持续更新。

5. **🔧 更广泛的非功能性 Bug 修复：** 将"主动修复"范式推广到其他类型的非功能性 bug（如性能问题、安全漏洞等）。

---

## 9 总结

本综述系统地回顾了 Web/JavaScript 内存泄漏检测与修复领域从 2013 年到 2025 年的技术发展。从 JSWhiz 的基础性探索，到 Madsen et al. 的事件调用图理论创新，再到 BLeak 的动态检测突破，以及 LeakPair 和 Shahoor et al. 的主动修复范式创新，我们看到了该领域从"只检测"到"检测+修复"、从"反应式"到"主动式"的清晰演进路径。

这些工作共同揭示了一个重要趋势：**对于特定类型的 bug，简单但针对性强的基于模式的方法可以比复杂的深度学习方法更可靠、更有效。** 未来，将基于模式的方法与 LLM 的代码理解能力相结合，可能是进一步提升内存泄漏自动化水平的关键方向。
